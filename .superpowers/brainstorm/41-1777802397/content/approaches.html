技术方案对比

AI 求职助手 — 三种全栈架构方案

方案 A — FastAPI + Vue.js（推荐）

Python 异步后端 + 现代 SPA 前端，前后端分离。FastAPI 原生支持流式响应，天然适合 AI/LLM 场景。Vue.js 在国内生态成熟，组件库（Element Plus）丰富。

Pros

  • FastAPI 异步 + 流式，AI 场景最佳
  • 前后端分离，部署灵活，可独立扩展
  • Vue 国内生态好，招人/上手都快
  • LangChain 官方支持 FastAPI 集成

Cons

  • 两套代码仓库，初始化工作量大一些
  • 需要处理 CORS / 认证跨域

方案 B — Django + htmx

Django 全栈单体应用，服务端渲染为主，htmx 做局部刷新。适合快速出 MVP，但前端交互体验不如 SPA。

Pros

  • Python 单一语言，开发效率高
  • Django Admin / ORM / Auth 开箱即用
  • 单体部署简单，Docker 单容器即可
  • 代码量大但结构清晰

Cons

  • AI 流式响应支持较弱（SSE 需额外处理）
  • 前端交互复杂时 htmx 吃力
  • 大型 SPA 级体验难以实现

方案 C — Next.js 全栈 + Python AI 微服务

Next.js 做全栈和 BFF，AI 处理抽成 Python 微服务。国内岗位更认，但如果后端重 AI，Python 的比例会越来越大。

Pros

  • SEO/SSR 开箱即用
  • 前后端一体化 TypeScript
  • React 生态最丰富

Cons

  • Python AI 代码变成"二等公民"
  • 两套服务之间维护成本高
  • 国内 Next.js 岗位少于 Vue

推荐：方案 A

理由：你的核心逻辑（AI 处理、RAG 检索、简历解析）都在 Python 侧，FastAPI 是 AI 项目的事实标准。Vue.js 在国内求职市场认可度高。前后端分离也符合展示"全栈能力"的目标。
