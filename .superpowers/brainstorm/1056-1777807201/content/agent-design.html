Agent 架构设计 — LangGraph + Agent SKILLS

从"函数调用"进化为"智能决策编排"

为什么用 LangGraph

当前设计是线性的：API → Service → LLM → 返回。LangGraph 让它变成有状态的智能决策流程：

当前：线性调用
POST /api/resumes/upload
→ parse() → return json

POST /api/jobs/search
→ match() → return list

每次调用独立，无上下文
目标：Agent 编排
用户上传简历
Agent 自主决策：
1. parse_resume ✓
2. 发现是 AI 方向 → build_profile
3. 自动搜索匹配职位
4. 发现匹配度低 → 建议优化
5. 推送相似经历鼓励

一次触发，Agent 自主编排全流程

Agent SKILLS 体系

每个 Skill 是 LangChain Tool，Agent 根据用户状态自主决定调用哪些 Skill：

🔧 工具型 Skills

parse_resume
PDF/Word/文本 → 结构化内容
build_profile
提取技能图谱+经验向量

针对目标JD生成优化版本

LangGraph 状态机设计

共享状态 (AgentState)： user_id, profile (画像), current_context (当前场景), conversation_history, pending_actions 核心图节点：
understand → 理解用户意图和当前状态
decide → Agent 决定调用哪些 Skill
execute → 执行选中的 Skill
observe → 评估执行结果，决定下一步
respond → 生成面向用户的自然语言响应 路由逻辑：
understand → decide → execute → observe → decide (循环) 或 observe → respond (完成)

我的意见

为什么应该做

  • LangGraph + LangChain 是同一个生态，集成零成本
  • Agent SKILLS 让功能模块化，新增能力只需注册新 Tool
  • 状态图让"贯穿全流程"的情感支持和画像更新变得自然——不再是各模块各自调用
  • 在简历上能体现出工程深度（Agent 架构 vs 简单的 API 调大模型）

需要考虑的

  • Agent 决策不可控时需要 fallback：关键路径（如简历解析）仍应有确定性调用路径
  • LangGraph 调试比普通 API 调用复杂，建议加 tracing（LangSmith 或自建）
  • 开发初期先做 3-4 个核心 Skill，验证模式后再扩展，不要一次做 10+ 个

建议：采纳。以 LangGraph StateGraph 为核心编排层，每个功能封装为 Tool/Skill。但保留 FastAPI 直接调用路径作为 fallback，关键操作不依赖 Agent 决策。
