设计反馈 — 三个问题的回应

1. 简历与 JD 匹配度评估

包含了，但在各个模块中分散呈现。更清晰的定位是：匹配度评估是一个跨模块的核心能力，被多个场景调用：

简历优化
上传简历后立即对目标 JD 评分，指出差距
职位匹配
批量匹配推荐职位，按匹配度排序
面试备战
基于匹配弱项生成针对性面试问题

2. "多版本管理"是什么

同一个用户可以为不同目标岗位维护不同版本的简历：

场景举例：
你是 Python 后端，投递时可能面对三类岗位：
• Python 后端工程师 → 侧重 Django/FastAPI、数据库优化
• AI 应用开发 → 侧重 LangChain、向量数据库、Prompt 工程
• 全栈工程师 → 侧重 Vue.js 项目经验、前后端协作

AI 基于同一份基础简历，为每个方向生成一个针对性的优化版本，不会互相覆盖。每个版本有自己的匹配评分历史。

3. 用户求职画像 — 我的建议

这个想法很好，应该加。我建议把它定位为系统的数据中枢，而不仅仅是"又一个功能模块"。

为什么值得加

  • 当前设计的问题：每个模块独立理解用户，简历解析一次、JD分析一次、面试又是一次 — 各模块对用户的认知是碎片化的
  • 画像的作用：一个结构化的用户画像（技能图谱 + 经验向量 + 偏好 + 目标），所有模块共享，越用越准
  • 情感支持的升级：有了画像，RAG 检索不再是泛泛的"后端求职者经历"，而是"2年Python、转行AI方向、投了50家还没上岸"这种精准匹配

画像应包含什么

简历提取（自动） — 技能标签、工作年限、教育背景、项目经验 → 向量
用户补充（主动） — 期望城市、薪资范围、目标行业、求职状态
行为积累（动态） — 投递偏好、高频目标岗位、面试薄弱环节
多维评分（自动） — 技能竞争力分、市场匹配度、简历完整度
