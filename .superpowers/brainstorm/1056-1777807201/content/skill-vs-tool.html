SKILL vs TOOL — 两层架构

修正理解：SKILL是业务资源，TOOL是原子操作

关键区别

TOOL（原子操作）
db_read() — 读数据库
db_write() — 写数据库
chroma_query() — 向量检索
call_llm() — 调用大模型
parse_pdf() — 解析PDF
send_notification() — 发通知

单一、不可再分的操作
LangChain @tool 实现
SKILL（业务资源集合）
score-match.md
optimize-resume.md
comfort-user.md

完成一项业务任务所需：
工作流程 + 执行指令 + 输出规范
.md 文件，Agent 按需加载

SKILL.md 格式

--- name: score-match description: 当用户想了解自己的简历与某个岗位的匹配程度时使用，或当系统中存在待匹配的职位-简历对时使用 --- # 简历-职位匹配度评估 ## 目标 评估用户简历与目标职位的匹配程度，给出多维度评分和可操作的改进建议。 ## 工作流程 1. 从用户画像中获取技能标签、工作年限、项目经验 2. 从目标职位中提取需求技能、经验要求、学历要求 3. 从四个维度对比：技能重合度、经验匹配度、学历匹配度、薪资匹配度 4. 生成评分报告，列出优势项和差距项 5. 对每个差距项给出具体改进建议 ## 输出格式 { "overall": float, "skill_match": float, "experience_match": float, "strengths": [...], "gaps": [...], "suggestions": [...] } ## 关联 - 匹配度低于0.6时，建议加载 optimize-resume 或 comfort-user

Agent 加载 SKILL 的流程

1. 启动时 — Agent 只拿到 SKILL 名称 + description 列表（极低 token）
2. 用户请求 — Agent 根据 description 判断需要哪个 SKILL
3. 加载 body — 读取对应 .md 全部内容，注入上下文
4. 执行任务 — Agent 按照 SKILL 中的工作流程，调用 TOOL 完成
5. 完成后 — 从上下文卸载 SKILL body，释放 token

完整技能清单

parse-resume.md
当用户上传或更新简历时，解析并提取结构化信息
build-profile.md
当简历解析完成或用户主动补充信息后，构建/更新求职画像
score-match.md
当用户想了解简历与岗位的匹配程度时使用
optimize-resume.md
当需要针对特定JD优化简历、或匹配度低于0.6时使用
match-jobs.md
当用户希望搜索/发现匹配的职位机会时使用
generate-interview-qs.md
当用户获得面试机会、需要针对性准备时使用
evaluate-answer.md
当用户完成模拟面试回答、需要评估和改进建议时使用
comfort-user.md
当用户遭遇被拒、低匹配、面试失败或需要鼓励时使用
onboard-user.md
当新用户首次使用系统，需要走完整初始化流程时使用
daily-checkin.md
当用户进行每日签到、或系统定时触发生成日报时使用
