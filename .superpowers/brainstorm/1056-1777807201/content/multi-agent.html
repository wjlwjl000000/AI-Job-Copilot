修正 1：去掉 StreamingMiddleware

你是对的，agent.astream() 原生支持流式输出，不需要中间件包装。去掉。

修正 2：多 Agent 划分方案

划分原则：上下文依赖 → 同一上下文的任务归一个 Agent → Agent 间仅传递结构化结果
Orchestrator Agent
SKILL: onboard-user
意图识别 → 任务分解 → 分发到专职 Agent → 汇总结果 → 生成响应
轻量协调，不做重型工作
Profile Agent
SKILL: parse-resume, build-profile
独立执行，无上游依赖
输出：结构化画像 → 传递给 Matching Agent
Matching Agent
SKILL: match-jobs, score-match, optimize-resume
依赖：Profile Agent 的输出
输出：匹配结果+弱项 → 传递给 Interview Agent
Interview Agent
SKILL: generate-interview-qs, evaluate-answer
依赖：Matching Agent 的弱项分析
输出：面试问题+评估 → 可直接返回前端
Support Agent
SKILL: comfort-user, daily-checkin
与主链路并行，Orchestrator 在任何节点检测到触发条件时可随时激活
输入：用户画像摘要 + 触发事件类型 → 输出：个性化鼓励内容

我的意见

为什么值得做

  • 每个 Agent 上下文独立，SKILL 数量少，决策更准，不会被其他 Agent 的 SKILL description 干扰
  • Profile/Support 可并行，减少用户等待时间
  • Agent 职责清晰，单 Agent 出问题不影响全局
  • 架构深度在简历上更有说服力

注意点

  • 5 个 Agent 意味着 5 套 system_prompt + 5 组 SKILL 分配，需要仔细调校
  • Orchestrator 是关键——它决定了任务分发的正确性。建议 Orchestrator 不执行重型 SKILL，只做路由和汇总

建议：采纳。5 个 Agent 分工明确，DAG 表达清晰。当前规模不需要更多 Agent，后续扩展时按上下文边界再拆分即可。
