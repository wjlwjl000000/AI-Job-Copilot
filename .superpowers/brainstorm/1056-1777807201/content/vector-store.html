向量存储方案调整

PostgreSQL + pgvector → Chroma

你的建议 vs 原方案

原方案：pgvector
PostgreSQL 中用 vector(1536) 字段
手写 SQL 做向量相似度查询
手动管理 embedding 生成和写入
一个数据库管所有
新方案：Chroma
LangChain Document 包裹文本
langchain_chroma 自动向量化+存储
.similarity_search() 一行检索
代码量减少 60%+

我的意见：采纳，理由如下

好处

  • LangChain Chroma 集成是官方维护的，from_documents() 一行搞定向量化+写入
  • 手写 pgvector 要管理 embedding 生成、维度对齐、相似度查询优化，LangChain-Chroma 全封装好了
  • Chroma 可以作为嵌入模式运行（无需额外服务），开发阶段零部署
  • 职责分离干净：PostgreSQL 管业务数据，Chroma 管语义检索

权衡

  • 多一个依赖。但 Chroma pip install 即可，部署时一个容器，代价很小
  • 生产环境 Chroma 建议独立部署，增加运维复杂度。但这个阶段不是问题

修订后的存储架构

PostgreSQL — 全部业务数据
  UserProfile, Resume, Job, Application, Interview, User（用户表）

Chroma — 三个 Collection，纯向量检索
  jobs — JD 文本 → 语义匹配职位
  stories — 经历故事 → RAG 情感支持
  profiles — 用户画像摘要 → 相似用户匹配

数据实体调整

从原实体中去掉 embedding 字段，保持纯业务属性：

UserProfile — 不再存 vector，PostgreSQL 只存结构化数据 skill_tags, work_years, education, projects, target, preference, scores Job — 去掉 embedding 字段 source, jd_content, requirements, status ExperienceStory — 去掉 embedding 字段 tags, content, source, is_anonymous 向量全部由 Chroma 管理，通过 Document.metadata 关联回 PostgreSQL ID
